This page intentionally reads like a prototype board. The first public Windows package appears only after real devices, sessions, volume controls, and unsupported states pass release gates.

Prototype layers, not a finished product.

Each layer is shown as a checkpoint: prove the public controls, spike app routing, then decide whether a signed driver/service is worth shipping.

Compatibility notes
Layer 1

Public Windows controls

Real endpoint list, endpoint volume and mute, audio sessions, per-session gain, and selected-source shortcuts.

Layer 2

Experimental app routing

Process-loopback capture, render-to-endpoint experiments, source sample-rate display, and duplicate-playback control.

Layer 3

Production backend

Signed virtual endpoint driver or service for reliable app-as-input devices, group sync, and lower-latency fan-out.

Roadmap before release.

The Windows version has its own tray app, C++/WinRT backend, and release gates before a Windows ZIP or MSIX appears here.

Roadmap status

Build the public controls first, then graduate routing into a real Windows audio backend.

The first Windows release should be useful even before driver-level routing exists: list devices, control endpoint volume, show audio sessions, and make unsupported routing states honest.

01 Scaffolded

Native Windows foundation

Create a sibling Windows app instead of trying to port SwiftUI and Core Audio directly.

  • WinUI 3 dashboard shell
  • C++/WinRT audio backend boundary
  • Shared route/setup JSON contracts
02 Scaffolded

Public audio controls

Use supported Windows Core Audio APIs for controls that can work reliably right away.

  • MMDevice endpoint discovery
  • Endpoint volume and mute
  • Audio session list and per-session gain
03 Next

Tray app and quick router

Match the Mac menu-bar workflow with a Windows tray flyout and a full dashboard window.

  • Tray quick routes
  • Real process icons
  • Selected-source volume shortcuts
04 Next

VM and hardware QA

Prove the app on Windows 11 before a public package is linked from the website.

  • Windows 11 VM smoke tests
  • Bluetooth, USB, HDMI, virtual outputs
  • Spotify, Chrome, Zoom, and system sessions
05 Experimental

Process-loopback routing

Investigate app capture and render-to-device routing with Windows process loopback where supported.

  • Capture selected process audio
  • Render to a chosen endpoint
  • Show Live, Saved, or Requires Driver honestly
06 Driver path

Production routing backend

Reliable SoundSource-style routing on Windows may need a service or signed virtual audio endpoint driver.

  • Signed driver or service installer
  • App-as-input devices for mixers
  • Group Play sync and latency controls

Before the first Windows download

Build passes on Windows runner Tray and dashboard open cleanly Real devices and sessions appear Volume and mute work where supported Unsupported routing is labeled clearly ZIP or MSIX install path is documented

Windows QA matrix

  • Windows 11 current stable
  • Built-in speakers, Bluetooth, USB, HDMI, and virtual outputs
  • Spotify, Chrome, Apple Music or iTunes, Zoom, and system sessions
  • Device connect and disconnect while audio is playing
  • High contrast, keyboard navigation, and reduced motion
